目标：防止项目换了一台电脑或重命名了文件夹后出现找不到模型、材质或配置的情况。
方法：使用脚本或命令全局搜索 usv_sim_full 原码目录下所有 .xacro、.urdf、.world、.sdf 以及 .py (生成器脚本) 和 .yaml 文件。

期望结果：不能有任何一条实质性业务代码包含绝对路径（注释或 README 等说明文档除外）。
### 第二步：排查越界 ROS 包引用 (耦合错误)
目标：确保我们的船体、推进器、传感器等资产实现了物理隔离，不再依赖 wamv_description 或 vrx_gazebo 里的模型文件。
方法：搜索所有的 package:// 协议路径。
黑名单关键字：package://wamv_description、package://vrx_...。
期望结果：所有的 ROS 包级别的资产引用，必须且只能是 package://usv_sim_full/...。
### 第三步：排查非法的 Gazebo 模型引用 (潜在隐患)
目标：Gazebo 使用 model:// 协议来加载环境和模型。我们需要确保我们自定义的船体和局部模型是基于当前包内资产加载的。
方法：搜索所有的 model:// 协议路径。
分类判定：
合法的业务引用：model://usv_sim_full/... （我们在配置中暴露的专属模型路径）。
合法的公共环境引用：model://ocean_waves 或 model://short_navigation_course_0 等 Gazebo 官方/公共环境资源。
违规的依赖引用：例如 model://WAM-V-Base/...（这说明依然借用了外部的 WAM-V 模型库，没有做到自有资产包裹）。
### 第四步：核查生成器脚本的拼接逻辑 (动态生成检查)
目标：因为我们的项目有通过 Python (session_manager.py 等) 动态生成 .urdf 的环节，要确保动态生成的中间文件里使用的也是干净的路径。
方法：人工+工具审阅生成器代码中写死的模板字符串，确保它没有在代码层面上向生成的配置中注入外部依赖（如 wamv_description）。
