## rviz map坐标报错
在目前的代码仓库中，无人船产生“真实全局位置”的底层是 Gazebo 的 OdometryPublisher 插件。这个插件的配置位于：
📌 wamv_p3d.xacro
src/vrx/vrx_urdf/wamv_gazebo/urdf/components/wamv_p3d.xacro
其核心代码写着：
```c++
<plugin filename="gz-sim-odometry-publisher-system" name="gz::sim::systems::OdometryPublisher">
    <odom_frame>map</odom_frame>  <!-- 就在这里！把坐标系强行命名为 map -->
    <robot_base_frame>$(arg namespace)/base_link</robot_base_frame>
</plugin>
```
这导致 Gazebo 往外扔的里程计数据（话题 /model/usv_sim_full/odometry）的表头自带了 frame_id='map' 的标签。

当 RViz 接收到这个数据，试图画出船的轨迹时：
    RViz 的全局视野（Fixed Frame）现在是在 usv_sim_full/base_link。
    RViz 发现传来的里程计数据属于 map。
    RViz 去问 TF 系统：“map 到 usv_sim_full/base_link 的相对位置怎么转？”
    TF 回答：“没有人在广播 map 转换，也就是你看到的 frames 中没有它”。
    结果： RViz 无法处理这条消息，直接将其塞进死胡同（缓冲区），等满了之后就开始疯狂抛出 "discarding message because the queue is full" 的报错。
### 解决方法
1. 关闭订阅
在打开的 RViz2 界面左侧 Displays 面板中。
找到 Odometry 这一项（前面有个打勾的小框）。
把 Odometry 前面的勾取消掉。
原理：只要 RViz 不再去订阅和试图渲染这个里程计，报错瞬间就会停止。

2. 发布map指令
在终端中运行以下命令，手动发布 map 到 usv_sim_full/base_link 的转换：
```bash
ros2 run tf2_ros static_transform_publisher 0 0 0 0 0 0 map usv_sim_full/base_link
```
这将创建一个静态转换，使 RViz 能够正确处理里程计数据。


## 推进器的控制问题

在当前的仿真环境中，推进器无法响应 ROS 中发布的控制命令。虽然 ROS 侧的话题存在并且能够接受到数据，但 Gazebo 中的推进器没有任何反应。这表明 ROS 和 Gazebo 之间的通信存在问题，可能是桥接器配置不正确或者消息类型不匹配导致的。
1. 检查 ROS 和 Gazebo 桥接器配置
确实是桥接配置出错,
在之前的 session_manager.py 里：
指定将 /{ns}/thrusters/left/thrust 桥接到了：
gz_topic_name: /model/usv_sim_full/joint/left_engine_propeller_joint/cmd_vel
但实际上在 Gazebo Harmonic 中，gz::sim::systems::Thruster 物理推进器插件不听 cmd_vel。它会自动生成并监听 /{namespace}/thrusters/{name}/thrust。信号一直被发到了无人订阅的 cmd_vel 废弃频道，所以船纹丝不动！

我已经完成了以下修复：

我修改了 session_manager.py，将推力与角度指令的 gz_topic_name 分别纠正为 Gazebo 实际监听的 /{sanitized_bridge_ns}/thrusters/left/thrust 和 /{sanitized_bridge_ns}/thrusters/left/pos（右侧同理）。
