
🎯 1. 项目核心愿景 (Core Vision)
构建一个 基于 ROS 2 + Gazebo Garden 的、高度可配置的、数据驱动的水面无人船仿真平台。

它不仅仅是一个“把船跑起来”的演示，而是一个算法验证台。

对内：通过 YAML 配置即可改变船体物理特性（质量、水阻）和传感器布局，无需修改 C++/URDF 源码。

对外：提供标准的 ROS 2 接口，支持上层算法（导航 Nav2、定位 EKF、感知 CV）的快速迭代与压力测试。


📝 2. 具体目标细节 (Detailed Objectives)
我们将目标拆解为四个维度的具体指标：

A. 资产与依赖管理 (Assets & Dependency)
双轨制架构：

依赖层 (src/vrx)：保留 VRX 官方源码，负责编译生成物理插件 (.so)。我们不修改它，确保物理引擎的原生性和可升级性。

业务层 (src/usv_sim_full)：克隆并拥有独立的 WAM-V 模型资产（Mesh, Materials, Xacro）。所有修改（如换个颜色、加个支架）仅发生在这里。

资源闭环：业务包内的所有文件引用必须闭环，即只引用 package://usv_sim_full/...，杜绝跨包引用导致的路径混乱。

B. 动态实体构建 (Dynamic Entity Construction)YAML 驱动：用户只需编辑 config/full_config.yaml。Python 生成器 (Session Manager)：不直接拼接 XML 字符串（拒绝硬编码）。采用 Xacro Wrapper 模式：Python 读取 YAML -> 生成一个临时的 wrapper.xacro -> 该文件调用底层的 wamv_base.xacro 并传入参数。可配参数：物理：质量 ($m$)、惯性矩阵 ($I_{xx}, I_{yy}...$)、水动力系数 ($X_u, X_{uu}...$)。传感器：类型（Lidar, Camera, IMU, GPS）、安装位置 $(x,y,z, r,p,y)$、ROS 话题名称。

C. 环境与交互 (Environment & Interaction)
程序化场景：

支持 固定列表（在指定坐标生成浮标）。

支持 随机区域（在矩形区域内随机撒布障碍物，用于测试避障算法）。

通信桥接 (Bridge)：

自动生成 ros_gz_bridge 配置。

当你在 YAML 里增加一个激光雷达，系统启动时自动建立 /lidar/points 的桥接，无需人工修改 Launch 文件。

D. 扩展性接口 (Extensibility Interfaces)
推进器布局：目前实现 双尾推 (Differential)，但数据结构上预留 全向推 (Omni) 的接口。

算法预留：Launch 文件中必须预留 Nav2 和 Robot_Localization 的启动入口（IncludeLaunchDescription），即使现在不写，位置要留好。
